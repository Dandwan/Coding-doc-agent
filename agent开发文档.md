# Agent开发文档：需求澄清与文档生成Agent（项目管理版）

## 1. 项目概述

本项目旨在开发一个智能Agent（DocAgent），通过Web前端与用户交互，以多轮选项式提问的方式逐步澄清用户关于“待开发工具”的需求，最终生成一份结构完整、内容详实的Markdown格式开发文档（即“Agent开发文档”）。该文档将作为后续实际开发工作的依据。

DocAgent采用**项目管理**模式：用户可创建或打开项目，每个项目独立管理其对话历史、生成的Agent开发文档版本，并可关联一个“项目开发文档”作为参考。DocAgent支持全局配置项目文件夹位置、API接口参数（GPT兼容格式）以及文档存放规则，实现轻量、灵活的开发辅助工具。

**重要设计原则**：
- 尊重已有项目：当用户指定一个已存在的文件夹作为项目根目录时，DocAgent不会删除或修改文件夹中已有的非相关文件，仅按配置添加必要的元数据文件和文档版本。
- 容错性：即使项目开发文档不存在，DocAgent也能正常工作（跳过加载或友好提示），待后续开发Agent生成后即可读取。

## 2. 该项目的所有功能

### 2.1 项目管理与主页面
- **项目列表**：进入应用后显示主页面，列出所有已创建的项目（名称、最后更新时间）。
- **创建项目**：用户可创建新项目，输入项目名称，并**指定项目文件夹路径**（可以是新文件夹或已存在的文件夹）。工具会在该文件夹中按配置规范存放Agent开发文档和读取项目开发文档。
  - 若文件夹已存在，DocAgent会检查其中是否已有`meta.json`，若无则创建，但不会影响其他已有文件。
- **打开项目**：点击项目进入项目工作区，加载该项目的历史会话、已生成的Agent开发文档版本等。
- **项目设置**：每个项目可单独配置其项目开发文档的路径（或使用全局默认）。

### 2.2 全局设置
- **项目根目录**：设置所有项目的默认存放位置（如 `~/DocAgentProjects`），应用会记住该路径。创建项目时可选择使用该根目录下的子文件夹，也可指定其他路径。
- **API配置**：设置GPT兼容的API URL、API Key、模型名称（支持自定义端点）。
- **文档存放规则**：
  - 指定项目开发文档的固定文件名（如 `PROJECT.md`）及存放路径（相对于项目根目录）。
  - 指定Agent开发文档的存放目录及命名规则（例如按日期/时间生成版本文件）。
- **配置持久化**：所有全局设置保存在本地（如用户目录下的配置文件），应用启动时加载。

### 2.3 项目工作区
项目工作区包含以下子功能：

#### 2.3.1 对话历史管理（多会话）
- **每个项目独立管理其对话会话**：左侧面板显示当前项目下的所有对话会话（会话名称、最后更新时间）。
- 支持新建会话、重命名会话、删除会话。
- 每个会话独立存储对话记录、待解决问题清单、当前生成的Agent开发文档内容。

#### 2.3.2 选项式需求澄清
- **动态选项**：
  - 每个问题附带3-5个预设选项，最后一个选项根据用户选择状态动态显示：
    - 未选中任何预设选项时，显示“其他：”+输入框。
    - 选中任意预设选项后，变为“补充：”+输入框（可额外补充）。
  - 提供“跳过该问题并不再提出”按钮，选择后跳过该问题并在后续澄清中避免主动再次提出该问题（在会话中仍可通过新增输入或手动触发重新恢复讨论）。
- **确认提交**：用户必须至少选择一项或填写内容才能提交。
- **待解决问题清单**：实时展示当前尚未澄清的关键点。

#### 2.3.3 Agent开发文档管理
- **实时预览**：右侧面板展示当前会话最新生成的Agent开发文档（Markdown渲染）。
- **版本历史**：每次生成文档时，自动按日期时间（如 `2026-04-01_143022_DEVELOPMENT.md`）保存到项目文件夹下的指定目录（如 `docs/agent/`）。
- **文档对比/恢复**：支持查看历史版本，并可恢复到某个历史版本。

#### 2.3.4 项目开发文档参考
- 用户可在项目设置中指定项目开发文档的路径（如 `docs/project/PROJECT.md`）。
- DocAgent在对话时会尝试读取该文档内容作为系统提示词的一部分，用于理解已有设计，避免重复或冲突。
- **如果文档不存在，DocAgent正常继续**，仅提示用户该文档尚未创建，并照常进行需求澄清。后续开发Agent可生成该文档。
- **项目开发文档由开发项目的Agent维护**，DocAgent只读不写。因此，Agent开发文档中必须包含对项目开发文档的管理规范（见第6章），以便后续开发Agent能够正确维护该文档。

### 2.4 API集成与配置
- **GPT兼容格式**：使用OpenAI Chat Completions API格式，支持任何兼容端点（如本地vLLM、Ollama、代理等）。
- **轻量设计**：后端使用`requests`库进行API调用，无额外重型依赖。
- **动态切换**：API配置修改后立即生效，无需重启。

### 2.5 文件系统集成
- **打开项目文件夹**：在项目工作区提供一个按钮，点击可打开系统文件管理器定位到项目根目录。
- **修改项目文件夹**：允许用户更改当前项目的存放路径（需移动已有文件，且不会影响其他项目）。
- **全局默认路径**：在设置中可修改全局项目根目录，新建项目时默认在该目录下创建子文件夹。

## 3. 细节要求

### 3.1 技术栈
- **后端**：FastAPI（轻量），使用`uvicorn`运行，所有状态存于内存（会话、项目）或文件系统（配置、文档）。
- **前端**：HTML/CSS/JS，使用Fetch API，Markdown渲染`marked.js`。
- **文件存储**：JSON格式存储项目元数据（项目列表、会话列表），Markdown存储文档内容。
- **API调用**：使用`requests`库，支持超时、重试、错误处理。

### 3.2 全局配置存储
- 配置文件位置：`~/.docagent/config.json`（跨平台兼容）。
- 内容示例：
  ```json
  {
    "projects_root": "/home/user/DocAgentProjects",
    "api": {
      "url": "https://api.openai.com/v1/chat/completions",
      "api_key": "sk-xxx",
      "model": "gpt-4",
      "temperature": 0.7,
      "timeout": 60
    },
    "doc_paths": {
      "project_doc": "docs/project/PROJECT.md",
      "agent_doc_dir": "docs/agent"
    }
  }
  ```

### 3.3 项目结构与元数据
每个项目在其指定的文件夹下（由用户在创建时指定）应有以下结构：
- `meta.json`：项目元数据，包括项目名称、创建时间、最后打开时间、项目开发文档路径（可覆盖全局）。
- `sessions/`：存放各会话的JSON文件，每个文件包含对话历史、待解决问题清单、当前文档内容等。
- `docs/`：按照配置存放项目开发文档和Agent开发文档历史版本（如果配置的路径在项目文件夹内）。
- **注意**：DocAgent不会删除或修改上述目录外的任何已有文件。

### 3.4 项目开发文档不存在时的处理
- 在加载项目时，DocAgent会尝试读取配置中指定的项目开发文档路径。
- 若文件不存在，系统会记录一个标志（例如`project_doc_exists = False`），并在对话的系统提示词中说明“尚未提供项目开发文档，后续开发Agent将生成它”。
- 前端可在项目设置中提示用户文档缺失，但不影响DocAgent的正常对话和文档生成功能。

### 3.5 Agent开发文档版本管理
- 每次生成文档时，自动保存到`{agent_doc_dir}/{YYYY-MM-DD_HHMMSS}_DEVELOPMENT.md`。
- 同时保留最新一份为`DEVELOPMENT.md`（便于快速访问）。
- 前端展示所有历史版本列表，支持下载和恢复。

### 3.6 交互与体验
- **主页面**：简单卡片式项目列表，提供“新建项目”按钮，新建时需输入项目名称并选择或输入项目文件夹路径。
- **项目工作区**：采用三栏布局（左侧会话列表，中间对话区，右侧文档预览区），顶部工具栏包含项目设置、打开文件夹、全局设置等入口。
- **动态选项**：使用JavaScript监听选项变化，实时切换“其他/补充”状态。

- **AI 调用可视反馈**：在所有与 AI 交互的环节（需求澄清、选项生成、文档生成等），前端会显示统一的处理状态：
  - 固定全局状态条：页面右上角展示当前 AI 请求状态、进度条与百分比、状态文字（例如“AI思考中...”）；成功/失败会有不同样式提示。
  - 加载动画与进度：显示旋转图标和渐进式进度条（自动前进到一定上限，真实进度由前端根据后端阶段回调更新）。
  - 禁用关键操作：在 AI 处理中会临时禁用“确认提交/跳过/保存对话”等主要按钮，并显示对应的处理中文案，避免重复提交或冲突操作。

- **上下文保留策略**：系统在构建每次 AI 请求的提示词时，会聚合该项目下的所有会话历史（包括当前会话与其他会话的历史）作为项目级上下文，以便跨会话信息被保留与复用，改善长期回顾与多通路需求收敛的效果。

### 3.7 错误处理
- 对API调用失败、文件读写错误等提供友好提示。
- 会话状态定期自动保存（如每次提交后）。

### 4.2 核心模块实现方式

#### 4.2.1 `backend/api/llm_client.py`
- 使用`requests`库实现，封装`post`请求。
- 支持超时、重试（简单循环）。
- 示例方法：
  ```python
  def get_response(self, messages: list) -> str:
      headers = {
          "Authorization": f"Bearer {self.api_key}",
          "Content-Type": "application/json"
      }
      payload = {
          "model": self.model,
          "messages": messages,
          "temperature": self.temperature
      }
      response = requests.post(self.url, headers=headers, json=payload, timeout=self.timeout)
      response.raise_for_status()
      return response.json()["choices"][0]["message"]["content"]
  ```

#### 4.2.2 `backend/project_manager.py`
- 管理所有项目：维护一个项目列表文件（如`~/.docagent/projects.json`），记录每个项目的路径和元数据。
- 创建项目时：
  - 若用户指定的文件夹不存在，则创建。
  - 若已存在，检查`meta.json`，如果不存在则创建基础元数据。
  - **不删除或修改任何其他已有文件**。
- 删除项目时仅从项目列表中移除，不删除文件夹（或可询问用户）。

#### 4.2.3 `backend/document/loader.py`
- 尝试读取项目开发文档。
- 若文件不存在，返回`None`并在日志中记录。
- 上层调用时根据返回值决定是否在系统提示词中包含文档内容。

#### 4.2.4 其他模块（会话管理、对话等）
- 与之前保持一致，但需传入项目开发文档加载状态。

### 4.3 API端点设计（更新）
| 端点 | 方法 | 描述 |
|------|------|------|
| `/api/config` | GET, POST | 获取/保存全局配置 |
| `/api/projects` | GET | 列出所有项目 |
| `/api/projects` | POST | 创建新项目（传入name，folder） |
| `/api/projects/{id}` | GET | 获取项目详情（含会话列表，及项目开发文档是否存在标志） |
| `/api/projects/{id}` | DELETE | 删除项目（仅从列表移除，可选删除文件夹） |
| `/api/projects/{id}/folder/open` | POST | 打开项目文件夹（触发系统文件管理器） |
| `/api/projects/{id}/sessions` | GET | 获取项目的所有会话 |
| `/api/projects/{id}/sessions` | POST | 创建新会话 |
| `/api/projects/{id}/sessions/{sid}` | GET | 获取会话详情（对话历史等） |
| `/api/projects/{id}/sessions/{sid}` | DELETE | 删除会话 |

## 6. Agent开发文档规范

Agent开发文档是由DocAgent生成的最终产物，它应作为开发后续项目的Agent（即“开发项目的Agent”）的主要依据。该文档必须包含以下部分，以确保开发Agent能够准确理解并实施项目。

### 6.1 文档结构（必含部分）

Agent开发文档应采用Markdown格式，并至少包含以下三个一级标题：

#### 1. 项目功能清单
- 以列表形式清晰列出所有功能点。
- 每个功能点应包含简要描述和预期行为。
- 如有必要，可按模块分组。

#### 2. 项目细节
- **用户交互细节**：命令行参数、API端点定义（方法、路径、请求/响应示例）、界面元素等。
- **业务流程细节**：关键业务逻辑的步骤描述、异常处理流程、数据流向。
- **非功能需求细节**：性能指标（如响应时间、并发量）、安全性要求（如认证方式、加密）、可扩展性要求。
- **依赖与环境**：所需的第三方库、运行环境要求（如Python版本、操作系统）。

#### 3. 代码架构与实现方式
- **技术栈选型**：明确使用的框架、库、数据库、中间件等。
- **目录结构**：提供标准化的项目文件结构，并说明每个目录/文件的职责。
- **核心模块设计**：描述主要类、函数及其职责，可使用类图或序列图辅助说明。
- **数据模型**：定义数据库模型或核心数据结构（如JSON Schema）。
- **接口定义**：详细定义API或内部接口的输入输出格式。

### 6.2 项目开发文档管理规范

为了确保DocAgent与开发项目的Agent能够高效协同，Agent开发文档中**必须包含**对项目开发文档的管理规范，具体内容如下：

#### 6.2.1 项目开发文档的存放位置
- 项目开发文档应存放在项目文件夹下的固定位置，由全局设置中的`doc_paths.project_doc`指定（默认路径为`docs/project/PROJECT.md`）。
- DocAgent在运行时将从该位置读取文档内容作为上下文；开发项目的Agent在更新项目开发文档时，必须将更新后的内容写回该文件。

#### 6.2.2 项目开发文档的维护责任
- 项目开发文档由**开发项目的Agent**负责维护。当开发项目的Agent根据Agent开发文档完成工具开发后，应当同步更新项目开发文档，记录最终的实现细节、API变更、已知问题等。
- DocAgent不对项目开发文档进行修改，只读取其内容作为需求澄清的背景。

#### 6.2.3 项目开发文档的内容建议
- 建议项目开发文档遵循与Agent开发文档类似的结构，但更侧重于**实现记录**和**运维信息**，例如：
  - 已实现功能列表（与Agent开发文档对比，标记已完成项）
  - 部署说明
  - 配置项说明
  - 测试用例与结果
  - 待办事项
- 项目开发文档可以包含Agent开发文档的引用（如“详见`docs/agent/2026-04-01_143022_DEVELOPMENT.md`”），以避免重复内容。

#### 6.2.4 协同工作流
1. **需求阶段**：用户使用DocAgent生成Agent开发文档（版本v1）。
2. **开发阶段**：开发项目的Agent阅读Agent开发文档（v1），进行编码实现，并实时更新项目开发文档，记录开发过程中的决策和调整。
3. **迭代阶段**：当需求变更时，用户可再次使用DocAgent，加载当前的项目开发文档作为上下文，生成新的Agent开发文档（v2）。开发项目的Agent再基于v2进行增量开发，并继续更新项目开发文档。

通过以上规范，DocAgent和开发项目的Agent形成闭环，确保需求与实现始终保持同步。

## 7. 后续工作：更新本开发文档

本开发文档（即当前文档）描述的是DocAgent自身的开发过程。当DocAgent功能升级或需求变更时，开发人员可以运行DocAgent，选择“更新开发文档”功能，DocAgent将：
1. 加载当前的`DEVELOPMENT.md`（即本文档）。
2. 通过对话澄清需要新增或修改的功能点。
3. 生成更新后的开发文档，并保存为新的版本。

这样，DocAgent自身实现了文档的自我维护，形成闭环。

## 8. 部署与发布

完成DocAgent的开发后，需要将工具上传至GitHub仓库，并创建conda虚拟环境以便部署和运行。

### 8.1 上传至GitHub
1. 在本地项目根目录初始化Git仓库（如果尚未初始化）：
   ```bash
   git init
   ```
2. 添加远程仓库地址：
   ```bash
   git remote add origin https://github.com/Dandwan/Coding-doc-agent.git
   ```
3. 添加所有文件并提交：
   ```bash
   git add .
   git commit -m "Initial commit: DocAgent v7.0"
   ```
4. 推送到远程仓库：
   ```bash
   git branch -M main
   git push -u origin main
   ```

### 8.2 创建Conda虚拟环境
为了隔离依赖，建议使用conda创建虚拟环境并安装pip依赖：

1. 创建名为`Coding-doc-agent`的conda环境（Python 3.9+）：
   ```bash
   conda create -n Coding-doc-agent python=3.9
   ```
2. 激活环境：
   ```bash
   conda activate Coding-doc-agent
   ```
3. 安装项目依赖（使用`requirements.txt`）：
   ```bash
   pip install -r requirements.txt
   ```
4. 运行DocAgent：
   ```bash
   uvicorn backend.main:app --reload
   ```
   访问 `http://localhost:8000` 即可使用。

### 8.3 环境变量配置
在运行前，确保已配置必要的环境变量（如API密钥），可通过`.env`文件或系统环境变量设置。参考`.env.example`。

### 8.4 生产部署建议
- 使用`gunicorn` + `uvicorn` workers 提高并发能力。
- 将静态文件交由Nginx托管。
- 使用Redis或数据库替代内存存储会话（根据实际需求）。
